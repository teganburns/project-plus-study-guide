    <p class="quiz-meta">Chapter 5 Question 1</p>
    <h2>Which one of the following is NOT true about the project plan?</h2>

    <ol class="quiz-options" type="A">
      <li>It contains baselines for reference when measuring progress</li>
      <li>It contains lots of subsidiary plans</li>
      <li>It is the same as the project schedule</li>
      <li>It is sometimes known as the project management plan</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: C.</strong> The project plan, or project management plan, is broader than the project schedule. The schedule is one part of the planning information, not the same thing as the whole plan.</p>
      <ul>
        <li><strong>Baselines:</strong> True. The plan may include scope, schedule, and cost baselines used as reference points for measuring progress.</li>
        <li><strong>Subsidiary plans:</strong> True. The plan may contain many supporting plans, such as change, communications, risk, scope, schedule, and cost management plans.</li>
        <li><strong>Same as the project schedule:</strong> Not true. The schedule is only one component of the larger project management plan.</li>
        <li><strong>Project management plan:</strong> True. The project plan may be called the project management plan.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 2</p>
    <h2>Which is the best description of the Scope Statement?</h2>

    <ol class="quiz-options" type="A">
      <li>High-level requirements organized by stakeholder</li>
      <li>The description of the scope, major deliverables, and exclusions</li>
      <li>How the project work is broken down into bite-sized pieces</li>
      <li>Business needs to be addressed by the project</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: B.</strong> The scope statement describes what is included in the project, the major deliverables, and what is excluded.</p>
      <ul>
        <li><strong>High-level requirements by stakeholder:</strong> This sounds closer to early requirements analysis or stakeholder input, not the scope statement.</li>
        <li><strong>Scope, major deliverables, and exclusions:</strong> This is the best description of the scope statement.</li>
        <li><strong>Broken into bite-sized pieces:</strong> That points toward the WBS, which decomposes project work into smaller pieces.</li>
        <li><strong>Business needs:</strong> Business needs are more closely associated with the business case or the reason the project exists.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 3</p>
    <h2>What does MVP stand for?</h2>

    <ol class="quiz-options" type="A">
      <li>Minimum Viable Product</li>
      <li>Most Valued Professional</li>
      <li>Maximum Visible Project</li>
      <li>Mildly Vexing People</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: A.</strong> MVP stands for Minimum Viable Product: the smallest useful version of a product that can deliver value and support learning or feedback.</p>
      <ul>
        <li><strong>Minimum:</strong> It includes only what is needed for the product to be useful, not every possible feature.</li>
        <li><strong>Viable:</strong> It still has to work well enough to deliver value.</li>
        <li><strong>Product:</strong> The focus is on a usable product increment, not a full project plan.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 4</p>
    <h2>Which of the following are true regarding the Work Breakdown Structure? Choose three.</h2>

    <ol class="quiz-options" type="A">
      <li>It provides input to the project schedule</li>
      <li>It can be depicted as a hierarchical chart</li>
      <li>It forms part of the scope baseline</li>
      <li>It is only of value to the Project Manager</li>
      <li>It is an alternative way of illustrating the Project Charter</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answers: A, B, and C.</strong> The WBS decomposes project scope into a hierarchy of deliverables and work packages. It helps the team understand the work, supports communication, and provides input for later schedule planning.</p>
      <ul>
        <li><strong>Input to the schedule:</strong> True. The WBS helps define the work that later becomes activities in the project schedule.</li>
        <li><strong>Hierarchical chart:</strong> True. A WBS is commonly shown as a hierarchy, starting with the overall project and breaking down into deliverables and work packages.</li>
        <li><strong>Part of the scope baseline:</strong> True. The scope baseline includes the approved scope statement, WBS, and WBS dictionary.</li>
        <li><strong>Only valuable to the Project Manager:</strong> False. It is also useful for the team and stakeholders because it improves communication, transparency, and accountability.</li>
        <li><strong>Alternative Project Charter:</strong> False. The charter authorizes the project; the WBS breaks approved scope into deliverable work packages.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 5</p>
    <h2>The Project Sponsor wants all the product features in the fastest time. What needs to be adjusted in order to achieve this?</h2>

    <ol class="quiz-options" type="A">
      <li>The project budget</li>
      <li>The risk threshold</li>
      <li>The product scope</li>
      <li>The project business case</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: A.</strong> This is a triple-constraint question. If the sponsor wants full scope and faster delivery, the project will usually need more budget or resources to compensate.</p>
      <ul>
        <li><strong>Project budget:</strong> Correct. More resources or funding may be needed to deliver the same scope faster.</li>
        <li><strong>Risk threshold:</strong> This describes how much risk the organization is willing to accept; changing it does not directly create more time or capacity.</li>
        <li><strong>Product scope:</strong> Reducing scope could help go faster, but the question says the sponsor wants all product features.</li>
        <li><strong>Business case:</strong> The business case justifies whether the project should proceed; it is not the normal lever for delivering full scope faster.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 6</p>
    <h2>Once you have conducted a preliminary review, what is your next step in the change control process?</h2>

    <ol class="quiz-options" type="A">
      <li>Assess the impact of the change</li>
      <li>Implement the change</li>
      <li>Log the request</li>
      <li>Escalate to the decision maker</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: A.</strong> After the preliminary review, the next step is to assess the impact of the change.</p>
      <ul>
        <li><strong>Assess impact:</strong> Correct. This looks at effects on scope, schedule, cost, resources, risk, testing, and benefits.</li>
        <li><strong>Implement the change:</strong> Too soon. Implementation happens only after the change is approved and the plan is updated.</li>
        <li><strong>Log the request:</strong> This happens earlier, before the preliminary review.</li>
        <li><strong>Escalate to the decision maker:</strong> Decision makers are determined after the impact and recommendation are documented.</li>
      </ul>
      <p>Sequence cue: create or receive request -> log request -> preliminary review -> assess impact.</p>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 7</p>
    <h2>What does RTM stand for?</h2>

    <ol class="quiz-options" type="A">
      <li>"Release to Manufacturer," which provides a timeline for support of a new product</li>
      <li>"Right To Move," a data protection regulation for freedom of personal information transfer</li>
      <li>"Reason To Market," which provides a project-centric overview of the business benefits of each change</li>
      <li>"Requirements Traceability Matrix," to enable us to trace requirements back to the requestor and forward to work packages</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: D.</strong> RTM stands for Requirements Traceability Matrix.</p>
      <ul>
        <li><strong>Requirements Traceability Matrix:</strong> Tracks requirements from their source or requestor through to deliverables, work packages, and test criteria.</li>
        <li><strong>Release to Manufacturer / Right To Move / Reason To Market:</strong> These are distractors, not the Project+ meaning of RTM.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 8</p>
    <h2>The scope baseline includes which of the following? Choose three.</h2>

    <ol class="quiz-options" type="A">
      <li>The approved scope budget</li>
      <li>The Work Breakdown Structure (WBS)</li>
      <li>The approved scope statement</li>
      <li>The change control plan</li>
      <li>The project charter</li>
      <li>The WBS dictionary</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answers: B, C, and F.</strong> The scope baseline is made of the approved scope statement, the WBS, and the WBS dictionary.</p>
      <ul>
        <li><strong>WBS:</strong> Part of the scope baseline because it breaks scope into deliverable work packages.</li>
        <li><strong>Approved scope statement:</strong> Part of the scope baseline because it defines the approved scope, deliverables, and exclusions.</li>
        <li><strong>WBS dictionary:</strong> Part of the scope baseline because it provides supporting detail for WBS elements.</li>
        <li><strong>Approved scope budget:</strong> Not a scope baseline component. Cost information belongs with cost planning and cost baselines.</li>
        <li><strong>Change control plan:</strong> Explains how changes are controlled, but it is not part of the scope baseline itself.</li>
        <li><strong>Project charter:</strong> Authorizes the project to begin; it is not part of the scope baseline.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 9</p>
    <h2>How does an Agile environment manage product scope and changes?</h2>

    <ol class="quiz-options" type="A">
      <li>With the change framework</li>
      <li>With the sprint review</li>
      <li>With the Scrum team</li>
      <li>With the backlog</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: D.</strong> Agile teams usually manage product scope and changes through the backlog.</p>
      <ul>
        <li><strong>Backlog:</strong> Correct. Product scope is represented by backlog items, and change is handled through refinement, reprioritization, and iteration planning.</li>
        <li><strong>Sprint review:</strong> Useful for feedback, but it is not the main place where product scope is managed.</li>
        <li><strong>Scrum team:</strong> The team helps manage the work, but the backlog is the artifact used to manage scope and change.</li>
        <li><strong>Change framework:</strong> Too vague for this question. Agile relies on backlog management rather than the same heavy formal process used in predictive projects.</li>
      </ul>
    </details>
  </article>

    <p class="quiz-meta">Chapter 5 Question 10</p>
    <h2>Which is the best definition of the Work Breakdown Structure?</h2>

    <ol class="quiz-options" type="A">
      <li>The hierarchy of work packages and how they relate to the major deliverables</li>
      <li>The to-do list for the project team</li>
      <li>A matrix to enable us to trace requirements back to the requestor, and forward to work packages</li>
      <li>The chart which shows how work has been scheduled across multiple teams</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: A.</strong> The WBS is a hierarchy of deliverables and work packages.</p>
      <ul>
        <li><strong>Hierarchy of work packages:</strong> Correct. The WBS decomposes major deliverables into smaller work packages.</li>
        <li><strong>To-do list:</strong> Too task-focused. The WBS is deliverable-oriented, not a simple activity checklist.</li>
        <li><strong>Requirements matrix:</strong> This describes the RTM, not the WBS.</li>
        <li><strong>Scheduled across teams:</strong> This points toward scheduling or resource planning, not the WBS definition.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 11</p>
    <h2>If we want to produce something quickly and cheaply, what must we reduce in order to maintain quality?</h2>

    <ol class="quiz-options" type="A">
      <li>The performance criteria</li>
      <li>The budget</li>
      <li>The schedule</li>
      <li>The scope</li>
      <li>The CCB</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Answer: D.</strong> This is a triple-constraint question. If the project must be fast and cheap while maintaining quality, the usual trade-off is reducing scope.</p>
      <ul>
        <li><strong>Scope:</strong> Correct. Less scope can make it possible to deliver quickly and cheaply without sacrificing quality.</li>
        <li><strong>Budget:</strong> Reducing budget makes the "cheap" constraint even tighter.</li>
        <li><strong>Schedule:</strong> Reducing the schedule means going faster, which is already part of the scenario.</li>
        <li><strong>Performance criteria:</strong> Lowering quality criteria may harm quality, which the question says must be maintained.</li>
        <li><strong>CCB:</strong> The Change Control Board is a decision authority, not a triple-constraint lever.</li>
      </ul>
    </details>
  </article>

  <article class="quiz-card">
    <p class="quiz-meta">Chapter 5 Question 12</p>
    <h2>A project-specific change has been approved by the CCB. Put the following actions into the correct order.</h2>

    <ol class="quiz-options" type="A">
      <li>Implement the change</li>
      <li>Check the change is done right</li>
      <li>Make required updates to the project plan</li>
      <li>Let everyone know the change has been deployed in a project status meeting</li>
    </ol>

    <details class="quiz-answer">
      <summary>Show answer</summary>
      <p><strong>Order: C, A, B, D.</strong> After a change is approved, update the plan first, then implement the change, validate the implementation, and communicate deployment.</p>
      <ul>
        <li><strong>C. Update the project plan:</strong> Approved changes need to be reflected in affected plans, baselines, schedule, budget, scope statement, or other project documents before the work proceeds.</li>
        <li><strong>A. Implement the change:</strong> Do the approved work after the plan has been updated.</li>
        <li><strong>B. Check the change is done right:</strong> Validate that the change meets the requirements of the original change request.</li>
        <li><strong>D. Communicate deployment:</strong> Let stakeholders know the change has been deployed, often through normal status communications.</li>
      </ul>
    </details>
